# Image Refinement

The recipe for taking an existing image and producing a better one. Two flows depending on what the user actually wants — and the choice between them is most of the value, because they fail on different things.

## When to use

- The user has an image (any PNG/JPG path) and wants a different version of it.
- They invoke `/refine <path>`.
- They say things like "make this warmer", "remove the X", "tighter crop", "same thing but at golden hour", "iterate on this".

Don't use for: generating a brand-new image from scratch (use the `image-generation` skill / `/image`), building a theme (`/new-theme`), or batch operations across many images.

## First-run check (do this before anything else)

Check the onboarded marker at `$env:USERPROFILE\.proofsheet\onboarded` (Windows) or `~/.proofsheet/onboarded` (Unix), plus the relevant API key for the provider on the sidecar (or both keys if no sidecar). Branch per the table in the `proofsheet-onboarding` skill:

- Marker missing + keys missing → strong nudge to run `/welcome`, do not proceed.
- Marker missing + keys set → soft prompt: "First time using proofsheet? Type `tour` for `/welcome`, or `skip` to dispatch your refinement. I'll only ask once."
- Marker exists → just proceed (key-missing halts use standard instructions, no welcome suggestion).

## Prerequisites

Refinement supports both providers via `--provider gemini|openai`. The provider defaults to whatever the original image was generated with (read from the sidecar's `provider` field). If the sidecar is missing or has no provider field, default to `gemini`.

- For Gemini refinements: `GEMINI_API_KEY` env var set, with billing enabled. ~$0.04 per refinement.
- For OpenAI refinements: `OPENAI_API_KEY` env var set. Pay-per-image (~$0.04 standard, ~$0.17 high). The ChatGPT subscription does not cover this.
- `bin/gemini-image.js` and `bin/openai-image.js` both exist in the plugin and support text-to-image and image-to-image via flags.

If the required key is missing, halt with the same instructions as the image-generation skill.

## The flow

### 1. Read the image and its sidecar

The user invokes `/refine <path>`. Verify the file exists. Then look for the sidecar at `<dir>/.meta/<basename>.json` — e.g., for `generated/_images/foo.png`, the sidecar lives at `generated/_images/.meta/foo.png.json`. Note the `provider` field in the sidecar (`gemini` or `openai`); refinements default to the same provider unless the user explicitly overrides via `--provider <name>`:

- **If sidecar exists**: read and parse it. You now have the original prompt, ratio, theme (if any), timestamp, and model. This is gold — even if the image was generated 6 months ago, you can refine precisely.
- **If sidecar is missing** (e.g., user dropped in a foreign image or the sidecar got deleted): tell the user "No sidecar metadata for `<path>`. I can still refine it image-to-image, but I won't be able to tweak-the-original-prompt because I don't have it. Want to proceed image-to-image only, or paste the original prompt if you have it?"

### 2. Ask what kind of change

Show the user what you found in the sidecar (prompt, ratio, theme, timestamp), then ask:

> What kind of change do you want? Two options:
>
> 1. **Tweak the prompt and regenerate fresh.** Best for direction changes — "same idea but at golden hour", "make her older", "swap the medium to risograph", "different angle". Produces a *new* image, not an edit. Subject identity won't carry over.
>
> 2. **Modify this image directly (image-to-image).** Best for surgical edits — "warmer lighting", "remove the railing", "cleaner background", "fix the cropped foot". Keeps the same image, edits in place. Far higher fidelity for small fixes.

Wait for them to pick **1** or **2**. They can also ask for both ("regenerate with tweak first, then if it's good, image-to-image polish").

If they describe the change before picking a mode, infer which flow fits — major direction = (1), surgical fix = (2) — and confirm: "Sounds like image-to-image edit (mode 2). Confirm or override?"

### 3a. Tweak-and-regenerate path (mode 1)

Take the sidecar's original prompt as the starting point and apply the user's requested change. **Don't rewrite from scratch** — edit only the clause the user is changing.

Example:
- Sidecar prompt: "A weathered fisherman in a yellow oilskin, sitting at the end of a wooden pier, three-quarter angle, soft overcast morning light, shot on medium-format film, quiet contemplative mood, no text overlays, no logos, no watermarks, 3:2 aspect ratio."
- User says: "make it golden hour instead"
- New prompt: same as above but `soft overcast morning light` → `warm golden hour light from camera-left, late afternoon`.

Show the new full prompt to the user before dispatching so they can spot if you changed too much or too little.

Then dispatch the same way `image-generation` does:

```powershell
node "$env:CLAUDE_PLUGIN_ROOT/bin/<provider>-image.js" @'
<new prompt>
'@ "<new-output-path>" --ratio <ratio> --theme <theme-if-any>
```

Output path default: `./generated/_images/<original-slug>-refined-<YYYYMMDD-HHMMSS>.png` so refinements don't overwrite the original. Sidecar is written automatically.

### 3b. Image-to-image path (mode 2)

Take the user's edit instruction as-is (or refine it lightly for clarity) and pass it as the prompt, with the existing image via `--input`:

```powershell
node "$env:CLAUDE_PLUGIN_ROOT/bin/<provider>-image.js" @'
<edit instruction, e.g. "Warm up the overall lighting to a golden afternoon tone. Keep everything else identical — subject, composition, framing. No text overlays, no watermarks.">
'@ "<new-output-path>" --input "<original-path>"
```

Don't over-enrich the edit instruction. For image-to-image, less is more — the existing image carries the subject/composition/framing, the instruction only needs to describe the *change*. Bake in the standard negative cues (no text, no watermarks) so the model doesn't add them.

Output path default: `./generated/_images/<original-slug>-edited-<YYYYMMDD-HHMMSS>.png`. The sidecar records `inputPath` pointing to the original — so a chain of edits stays traceable.

### 4. Report

Print:
- Absolute path of the refined image
- Sidecar path
- File size
- The prompt or edit instruction that was sent
- For mode 1: a one-line diff vs. the original prompt ("changed: lighting clause"). For mode 2: the input path, so the user sees the chain.

Then offer: "another refinement?", "compare side-by-side?", "done." Don't auto-loop.

## When mode 1 vs mode 2 — the practical rule

**Mode 1 (tweak prompt, regenerate)** for changes where the *subject can change* and you only care about the direction:
- Different lighting / time of day
- Different medium ("now as a risograph")
- Different angle or framing
- Different mood
- Different setting / environment
- Aging or de-aging a person
- Swapping the subject for something else with similar properties

**Mode 2 (image-to-image)** for changes where you want to *keep this exact image* but fix something:
- Lighting tone adjustment (warmer / cooler / brighter / moodier)
- Removing a specific element ("remove the railing", "remove the watermark")
- Adding a specific element ("add fog in the background")
- Color correction
- Cleaning artifacts
- Tight crop or extension (with caveats — extension/outpainting works less reliably than internal edits)

If the user wants both (e.g., "regenerate at golden hour, then if good, remove the dock railing"), do them in sequence — mode 1 first, then mode 2 on the result.

## What actually works in image-to-image

- **Short, specific edit instructions.** "Warmer afternoon lighting" beats "make this look better in terms of lighting and mood."
- **Reference what to keep.** "Keep the subject, composition, and framing unchanged" reassures the model not to redraw everything.
- **Negative cues stay.** Always include "no text overlays, no watermarks" even on edits — otherwise the model sometimes adds them.
- **One change at a time is more reliable than five.** If the user wants three things changed, run three sequential edits rather than one mega-prompt.
- **Don't ask for impossible edits.** Image-to-image can't relight a scene from scratch, can't change physically impossible geometry, can't undo crops that lose information. If the user asks for something it'll fail at, say so before burning a generation.

## Edge cases

- **Foreign image (no sidecar).** Only mode 2 is available. Offer it. If the user pastes the original prompt manually, mode 1 becomes available too.
- **Sidecar exists but was edited by hand.** Treat it as truth. If the prompt field is empty, fall back to "no sidecar metadata."
- **Multi-step refinement chain.** Each refinement writes its own sidecar with `inputPath` pointing at the previous step. Users can `/refine` a refinement indefinitely.
- **Wrong output path.** If the user wants the refined image to overwrite the original, pass `--out <original-path>`. Warn them: "This will overwrite the original. Confirm?"
- **API errors.** Same handling as image-generation — don't auto-retry on quota/billing. Surface and ask.

## Don't

- Don't run mode 1 (regenerate) when the user wanted a surgical fix — they'll lose the original image's identity. When in doubt, ask.
- Don't run mode 2 (image-to-image) for major direction changes — the model will fight the existing image and produce a smushed result.
- Don't rewrite the original prompt from scratch in mode 1. Edit the specific clause the user is changing; preserve the rest.
- Don't skip the sidecar lookup. The original prompt is the whole point of the metadata system.
- Don't overwrite the original PNG without an explicit confirm.

## What success looks like

A user's "make this warmer" turns the existing image warmer without changing the subject, composition, or framing. A user's "same thing but at golden hour" produces a fresh image with golden-hour lighting, recognizably the same conceptual idea. Both outputs land with sidecars that let future refinements continue the chain.
